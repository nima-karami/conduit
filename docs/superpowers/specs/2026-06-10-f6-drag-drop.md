# F6 — Drag & drop: reorder sessions + tabs

## Goal
Let the user drag to reorder document tabs and session cards. Tab order is
in-memory (tabs are session-scoped, not persisted). Session order is persisted by
the host so it survives reload.

## Pure helper `src/reorder.ts` (unit-tested)
`moveBefore(ids: string[], dragId, targetId | null): string[]`
- remove dragId; insert it immediately before targetId; if targetId is null or not
  found, append to the end. No-op if dragId === targetId.

## Tabs
- DocTabs tabs become `draggable`; on drop, call `onReorder(dragId, targetId)`.
- docs reducer gains `{ type: 'reorder'; dragId; targetId }` applying moveBefore to
  the docs array (preserving activeId).
- Drop indicator (left border) on the hovered target.

## Sessions
- SessionItem becomes draggable; reordering is constrained to within the same
  project group (a session belongs to its repo; cross-repo moves are nonsensical).
- On drop, compute the new global session-id order and post `reorderSessions`.
- Host: `SessionManager.reorder(orderedIds)` rebuilds the session map in that order
  (unknown ids ignored, missing appended); groupByProject + list() already iterate
  in map order, so order is reflected and persisted via existing serialize.
- protocol: `WebviewToHost` += `{ type: 'reorderSessions'; order: string[] }`;
  main.ts handles → mgr.reorder → change → postState + persist.

## Acceptance criteria
1. Drag a tab left/right of another → tab order changes; active tab stays active.
2. Drag a session card within its group → order changes and persists across reload.
3. Cross-group session drop is rejected (no change).
4. Drop indicator shows where the item will land.
5. reorder unit tests (moveBefore: middle, ends, missing target, same id).
6. typecheck + build + tests green.
