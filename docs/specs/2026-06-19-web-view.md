# In-app web view — browse a webpage inside Conduit

## Problem

Conduit can open files, diffs, images, PDFs and the review surface as tabs, but a URL
always leaves the app (links route to the external browser via `shell.openExternal`).
There's no way to keep a webpage — docs, a localhost dev server, an agent's preview —
alongside the terminal and editor without context-switching to another window.

## Goal

Open an arbitrary `http(s)` URL as a **tab** in the center pane, with real browser
chrome (address bar, back/forward/reload/stop, loading + page title), rendered by an
embedded, isolated web view. Opt-in only — existing external-link behavior is unchanged.

## Architecture decision (conductor)

Render the page with Electron's **`<webview>` tag** embedded in the renderer DOM as a
new doc kind, **not** `WebContentsView`/`BrowserView`.

- Tabs render in the **renderer**; `<webview>` flows in the DOM exactly where the
  doc-view already mounts, so it inherits the tab/split/blur layout for free.
- `WebContentsView`/`BrowserView` are **main-process native overlays** positioned over
  a renderer rectangle — they would have to be manually tracked and repositioned over
  the active tab's bounds, fighting the existing renderer-driven layout and the
  `backdrop-filter` blur panels. Far more fragile for no benefit here.
- Trade-off accepted: `<webview>` is heavier per-instance and "soft-deprecated" but
  remains fully supported in Electron 42 and is the right fit for DOM-embedded tabs.

Requires `webviewTag: true` in the BrowserWindow `webPreferences` (`electron/main.ts`)
plus `will-attach-webview` hardening (below). This is a **new subsystem** built thin
and reversible on `git-run` (off `main`).

## Doc model

New `DocKind` value `'web'` (`webview/docs.ts`). A web doc's `path` is the **URL**, so
its id is `web:<url>` — reusing the existing id/ownership/persistence machinery with no
new state. Title is the page's `<title>` once loaded, falling back to the hostname.

- `titleOf` stays path-based for file/diff; the web title is set imperatively by the
  viewer via a new `onTitle(id, title)` callback that dispatches a doc rename (or a
  small `setDocTitle` reducer action), so the tab label tracks the live page title.

## Entry points

- **Command palette**: "Open Web Page…" → prompt for a URL (reuse the existing inline
  prompt/modal pattern), then `openWeb(url)` which dispatches `docs` `open` with kind
  `'web'`, owned by the active session (same ownership rule as files).
- No link hijacking: markdown/terminal links keep opening externally. (A future
  "Open in Conduit" affordance is out of scope.)

## URL handling — `webview/web-url.ts` (pure)

- `normalizeUrl(input): string | null`
  - trims; rejects empty.
  - if it already has an `http:`/`https:` scheme → return as-is (validated via `URL`).
  - if it parses as a host (`example.com`, `localhost:5173`) → prefix `https://`
    (localhost / `127.0.0.1` / a bare `host:port` may prefer `http://` — encode that
    rule here and unit-test it).
  - rejects any other scheme (`file:`, `javascript:`, `data:`) → `null`.
- `displayTitleForUrl(url): string` — hostname (+ path tail) fallback title.

Only `normalizeUrl`-approved `http(s)` URLs are ever set as the `<webview src>`.

## Browser chrome — `webview/components/web-view.tsx`

Layout: a toolbar row above a flex-filled `<webview>`.

Toolbar: Back, Forward, Reload/Stop (toggles on loading), an editable **address bar**
(shows current URL; Enter → `normalizeUrl` → navigate; invalid → inline shake/no-op),
and an external-open button ("open this URL in the system browser").

Behavior, wired to the `<webview>` element's API + DOM events:
- `did-start-loading` / `did-stop-loading` → loading state (Reload↔Stop, spinner).
- `page-title-updated` → update address-bar title + call `onTitle` (tab label).
- `did-navigate` / `did-navigate-in-page` → sync the address bar + can-go-back/forward
  (`canGoBack()`/`canGoForward()`).
- `did-fail-load` (top-level only, ignore aborted) → render an in-tab error panel
  (code + URL + Retry), never a blank frame.
- Back/Forward/Reload/Stop call `goBack()/goForward()/reload()/stop()`.
- Cleanup: listeners removed on unmount; no work after the element is gone.

`DocView` (`doc-view.tsx`) gains a `doc.kind === 'web'` branch → `<WebView url=…
onTitle=… />` (ordered before the file branch).

## Security hardening (host) — `electron/main.ts`

Enabling `webviewTag` widens the attack surface; lock the guest down:

- `app.on('web-contents-created')` / `win.webContents.on('will-attach-webview', (e,
  webPreferences, params))`:
  - delete `webPreferences.preload` (no Conduit preload in the guest),
  - force `webPreferences.nodeIntegration = false`, `contextIsolation = true`,
    `sandbox = true`,
  - if `params.src` is not an `http(s)` URL → `e.preventDefault()` (never attach a
    `file:`/other-scheme guest).
- The guest `webContents` gets its own `setWindowOpenHandler` → route `http(s)` to
  `shell.openExternal`, deny everything else (no popups escalate into chrome-less
  windows); block `will-navigate` to non-`http(s)`.
- `<webview>` element attributes: no `nodeintegration`, no `allowpopups`; `partition`
  is a dedicated persistent partition (`persist:webview`) so guest cookies/storage are
  isolated from the app origin.

## Acceptance criteria

- AC1: `normalizeUrl` — `'example.com'`→`'https://example.com'`; `'localhost:5173'`→
  `'http://localhost:5173'`; `'https://x.dev/a'` unchanged; `'file:///etc'`,
  `'javascript:alert(1)'`, `''` → `null`. (unit)
- AC2: Command palette "Open Web Page…" opens a `web:` tab owned by the active session;
  its label is the hostname until the page title loads. (unit on reducer/helpers)
- AC3: Real app — opening a web tab on a local `file://`-served fixture page renders the
  `<webview>` and shows the page; the address bar reflects the URL; the tab label
  becomes the page title. (e2e)
- AC4: Back is disabled at the first page and enables after an in-page navigation;
  Reload toggles to Stop while loading. (e2e best-effort; assert control state)
- AC5: A failed load shows the in-tab error panel, not a blank frame. (e2e on a bad URL)
- AC6: `will-attach-webview` strips preload + forces `nodeIntegration:false` /
  `sandbox:true`, and refuses a non-`http(s)` `src`. (unit on the pure hardening helper)
- AC7: External-link behavior elsewhere is unchanged; `npm run verify` green.

## Verification

- Unit: `test/unit/web-url.test.ts` (AC1), `test/unit/web-doc.test.ts` (AC2 reducer/
  title), `test/unit/webview-guard.test.ts` (AC6 — pure `hardenWebviewPrefs(prefs,
  src)` helper that `main.ts` calls from the event handler).
- Runtime e2e: `test/e2e/web-view.e2e.mjs` — serve a fixture page over `file://` (or a
  `data:`-free local html fixture loaded via a tiny `http` server on `127.0.0.1`),
  open a web tab, assert the `<webview>` element mounts, navigates, updates the address
  bar/title, and that a bad URL yields the error panel. Drives the **real built app**
  (webview tag only exists in Electron, not the browser preview).

## Decisions surfaced for the user (autonomous picks; reversible)

- D-1 **Renderer**: `<webview>` tag (picked) vs `WebContentsView` overlay. Picked for
  DOM-embedded tab fit; revisit only if `<webview>` is removed upstream.
- D-2 **Link hijacking**: keep external links opening in the system browser (picked) vs
  routing `http(s)` clicks into an in-app tab. Picked to avoid surprising the user;
  trivially reversible later.
- D-3 **Chrome depth**: full address bar + nav (picked, per "full feature-spec") vs a
  bare frame. No find-in-page / zoom / devtools / multi-history this pass.
