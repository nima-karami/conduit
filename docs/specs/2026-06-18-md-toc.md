# Document outline / table of contents

## Problem frame

**Job:** Long markdown docs (specs, ADRs, READMEs) are hard to navigate by scrolling.
A reader wants to see the document's structure at a glance and jump to a section —
and know where they currently are while reading.

**Actor:** anyone reading a multi-section `.md` in the rendered viewer.

**Success:** a toggleable outline panel lists the document's headings (h1–h4);
clicking an entry scrolls to that heading; as the user scrolls, the entry for the
current section is highlighted (scroll-spy). Short docs (few headings) don't show the
control — no clutter.

**Non-goals:** editing headings; a global project-wide outline; numbering; collapsing
subtrees; persisting open/closed across docs.

## Behavior & states

States: **unavailable** (doc has < 3 headings → no toggle shown) → **collapsed**
(an outline toggle button is visible, panel hidden) → **expanded** (panel shows the
heading list with the active section highlighted).

- **Toggle:** an outline button sits in the viewer's top-right control row next to
  "View source". Visible only when the doc has ≥ 3 headings (h1–h4).
- **Panel:** a floating, scrollable list anchored top-right inside the viewer; each
  entry is indented by heading depth (relative to the doc's shallowest heading).
  Clicking an entry smooth-scrolls its heading into view (reusing the existing slug
  ids + `scroll-margin-top`). The panel stays open after a jump.
- **Scroll-spy:** while scrolling the rendered view, the entry whose heading is the
  last one at/above the top of the viewport is marked active (highlighted). Computed
  from heading offsets in the scroll container; updates on scroll (throttled via rAF).
- **Doc switch:** outline rebuilds for the new doc; panel returns to collapsed (or
  stays per a simple local default — collapsed is fine).
- **Source view:** when "View source" is active, the outline is hidden (it describes
  the rendered structure).

## Data / interface contract

Pure, unit-testable core in `webview/md-toc.ts`:

- `interface TocEntry { id: string; text: string; level: number; depth: number }`
- `buildTocEntries(headings: { level: number; id: string; text: string }[]): TocEntry[]`
  — drops headings without an id/empty text; computes `depth = level - minLevel` for
  indentation. Returns [] for an empty/odd input.
- `pickActiveIndex(tops: number[], scrollTop: number, offset: number): number` — index
  of the last heading whose top ≤ `scrollTop + offset` (−1 before the first). Mirrors
  `findBlockForLine`'s contract; the scroll-spy selection logic, tested without a DOM.

Component `webview/components/markdown-toc.tsx`:
`{ entries: TocEntry[]; activeId: string | null; onJump: (id: string) => void;
   open: boolean }`. Presentational; no DOM scraping inside. (The open/close toggle
   lives in the viewer's control row, so the panel itself takes no `onToggle`.)

`MarkdownViewer` wiring:
- After render (and on doc change), scrape headings from the rendered container
  (`mdRef.current.querySelectorAll('h1,h2,h3,h4')`) → `{level,id,text}` → `buildTocEntries`.
  Reuses the ids the heading components already slugify; no re-parse of markdown.
- Scroll-spy: on the `.markdown` scroll container, read each heading's `offsetTop`,
  `pickActiveIndex` against `scrollTop`, set active id; throttle with rAF.
- Jump: `getElementById(id).scrollIntoView({ behavior:'smooth', block:'start' })`.

## Edge cases & failure modes

- **< 3 headings** → no toggle, no panel. (threshold avoids clutter on short docs)
- **Duplicate heading text** → ids are already de-duped by the existing `SlugFactory`
  (it suffixes collisions), so jump targets stay unique.
- **No headings / all h5–h6** → outline unavailable (only h1–h4 are listed).
- **Very long outline** → panel is independently scrollable, capped max-height.
- **Headings change after async content** → outline rebuilds on doc change; mermaid/
  math don't add headings, so no late mutation matters for MVP.
- **Reveal/flash (D7) + jump** → both use the same ids/scroll container; no conflict.
- **Reduced motion** → smooth scroll respects the OS setting via CSS where applicable;
  acceptable to keep smooth (matches existing link-jump behavior).

## Accessibility / i18n

- Panel is a `<nav aria-label="Document outline">` containing a list; entries are real
  `<button>`s (keyboard-focusable, Enter activates). The toggle is a labelled button
  with `aria-expanded`.
- Active entry marked with `aria-current="location"` (not color alone).
- No user-facing copy beyond "Outline"/aria-labels (English, consistent with app).

## Defaults vs. settings

No setting. Outline is available (when ≥3 headings) and starts collapsed. Rationale:
collapsed-by-default keeps the reading surface clean; a persisted preference is
divergent surface not worth it for MVP.

## Scope slicing

- **MVP:** toggle (≥3 headings), floating panel, indented entries, click-to-jump,
  scroll-spy active highlight, hidden in source view.
- **Out of scope:** subtree collapse, numbering, persistence, resizable panel,
  pinned/always-open mode.

## Acceptance criteria

- AC1: A doc with ≥3 headings shows an outline toggle; a doc with <3 headings does
  not. (playwright, two sample docs)
- AC2: Expanding the panel lists all h1–h4 headings, indented by depth. (playwright:
  entry count = heading count; indentation differs by level)
- AC3: Clicking an entry scrolls its heading into view. (playwright: assert the
  heading is near the top / scrollTop changed)
- AC4: Scrolling updates the active (highlighted, `aria-current`) entry to the
  current section. (playwright: scroll, assert active entry changes)
- AC5: `buildTocEntries` and `pickActiveIndex` behave per contract incl. empty/edge
  inputs. (unit)
- AC6: Source view hides the outline; `npm run verify` green; existing viewer
  behavior (links, reveal, copy) unaffected.

## Decisions Needed

none — collapsed-by-default + ≥3-heading threshold are conservative; DOM-scrape of
existing slug ids avoids a second markdown parse.
