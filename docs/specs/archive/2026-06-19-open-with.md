# Open with — Explorer context menu

## Problem

Conduit's file-tree row menu can open a file **inside** Conduit (editor/viewer) or
"Reveal in Explorer", but there is no way to hand the file to another application —
the everyday "I want to open this `.psd`/`.docx`/`.html` in its real app" action.
Right-clicking a file in the OS Explorer offers "Open" (default app) and "Open
with…" (app chooser); Conduit's Explorer should offer the same.

## Scope

Add two items to the **file-tree row** context menu (`right-pane.tsx` `openMenu`,
files only — not folders):

1. **Open externally** — open the file with its OS-default associated application
   (`shell.openPath`). The 80% case.
2. **Open with…** — open the native OS *application chooser* for the file, so the
   user picks which app. Windows: `rundll32.exe shell32.dll,OpenAs_RunDLL <path>`.
   On platforms with no CLI chooser primitive (mac/Linux), fall back to the default-app
   open so the item is never dead.

Out of scope: an in-app list of registered handlers (registry/`.desktop` parsing);
"Open with" on folders; remembering a per-extension app choice.

## Design

**Pure command builder** — `src/open-with.ts`:

```ts
export function openWithCommand(
  platform: NodeJS.Platform,
  filePath: string,
): { command: string; args: string[] } | null {
  if (platform === 'win32')
    return { command: 'rundll32.exe', args: ['shell32.dll,OpenAs_RunDLL', filePath] };
  return null; // no native chooser primitive — caller falls back to default-app open
}
```

Pure over its inputs → unit-tested without spawning anything.

**Protocol** (`src/protocol.ts`, `WebviewToHost`):
- `{ type: 'openExternalPath'; path: string }` — open with default app.
- `{ type: 'openWith'; path: string }` — open the OS chooser.

Both are read/launch-only over an arbitrary path the renderer can already read; no
write surface, so no workspace-root guard is required (mirrors `revealInExplorer`).

**Host** (`electron/main.ts`):
- `openExternalPath` → `void shell.openPath(m.path)`.
- `openWith` → `const cmd = openWithCommand(process.platform, m.path); if (cmd)
  spawnDetached(cmd.command, cmd.args); else void shell.openPath(m.path);` where
  `spawnDetached` uses `child_process.spawn(c, a, { detached: true, windowsHide:
  false }).unref()` (the chooser is a UI process that must outlive the IPC turn and
  show a window).

**Renderer** (`webview/components/right-pane.tsx`): in `openMenu`, for `node.kind ===
'file'`, after the existing "Open" item add (grouped under a separator with Reveal):
- "Open externally" → `post({ type: 'openExternalPath', path: node.path })`
- "Open with…" → `post({ type: 'openWith', path: node.path })`

`post` is already threaded into the component (used by reveal/copy).

## Security

- `openExternalPath`/`openWith` only ever pass a real filesystem path to
  `shell.openPath` / `rundll32 OpenAs_RunDLL` — never a URL or arbitrary string, so
  there is no scheme-injection / local-exec hazard beyond opening the file the user
  right-clicked (which they could already open via Reveal).
- The spawn is a fixed argv array (no shell string interpolation).

## Acceptance criteria

- AC1: `openWithCommand('win32', 'C:\\a\\b.txt')` → `{ command: 'rundll32.exe', args:
  ['shell32.dll,OpenAs_RunDLL', 'C:\\a\\b.txt'] }`; returns `null` for `'darwin'`/`'linux'`.
- AC2: The file-tree row menu shows **Open externally** and **Open with…** for a file;
  neither appears for a folder.
- AC3: Clicking **Open externally** posts `openExternalPath`; the host calls
  `shell.openPath(file)` (e2e: recorded via the existing `openPath` spy).
- AC4: Clicking **Open with…** posts `openWith`; on win32 the host builds the
  `OpenAs_RunDLL` command (unit) and spawns it. **The visible chooser dialog is
  needs-human-smoke** (a native dialog the smoke harness can't drive).
- AC5: `npm run verify` green.

## Verification

- Unit: `test/unit/open-with.test.ts` (AC1).
- Runtime e2e: `test/e2e/open-with.e2e.mjs` — open a project, trigger the menu's
  "Open externally" on a fixture file, assert `shell.openPath(file)` recorded. The
  "Open with…" win32 chooser is needs-human-smoke (manual: right-click → Open with…
  shows the Windows app picker).
