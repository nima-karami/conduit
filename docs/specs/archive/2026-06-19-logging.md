# Professional logging (leveled, file-backed, instrumented)

## Problem frame

**Job:** "When something misbehaves, let me see what the app actually did —
without rebuilding from source or attaching a debugger. Give me real logs I can
turn on, dial up, and hand over, so debugging isn't ghost-chasing."

Today logging is ad-hoc and ephemeral: scattered `console.log('[pty]', …)` /
`console.error` in the main process, a renderer `logToHost` that posts
`{ type: 'log', message }` to a thin host handler (`electron/main.ts:829`), and
nothing persisted. In a shipped (packaged) build there is no console to read and
no file to inspect — a user reporting "it broke" leaves no trail.

**Actors:** the developer (debugging locally), and the end user (who can flip
logging on and send a diagnostics bundle when reporting an issue).

**Success outcomes:**
- A single structured logger used across main **and** renderer, with standard
  levels and a per-scope tag.
- Logs persist to rotating files in the app's data dir, readable in a packaged
  build.
- The user can turn logging off/on and choose the level from Settings.
- The important seams are instrumented, so a repro produces a usable trail.
- A one-click way to reveal the logs (and bundle diagnostics) for a bug report.

**Non-goals (v1):**
- In-app live log viewer panel (deferred — see scope slices; user chose
  files + reveal for v1).
- Remote/telemetry shipping of logs off the machine.
- Logging raw terminal/PTY output streams (high-volume, sensitive — only
  lifecycle/▶events are logged, never the byte stream; see edge cases).

## Decisions (locked / assumed)

- **Surface = rotating files + a "Reveal logs" / "Copy diagnostics" action.** No
  in-app viewer in v1 (locked).
- **Levels:** `off` < `error` < `warn` < `info` < `debug` < `trace`. `off`
  silences everything (including the file sink). (locked set)
- **Control:** Settings → "Logging" group: an enable toggle + a level dropdown.
- **Default (assumed, reversible):** enabled at **`info`**. Rationale: a modest
  always-on trail is what makes a first bug report useful; `info` excludes the
  chatty `debug`/`trace`. Flag — could ship `warn` or off-by-default if preferred.
- **Storage (assumed):** `app.getPath('userData')/logs/`, files
  `conduit-<YYYYMMDD>.log` (or `main`/`renderer` split), **rotated** by size
  (e.g. 5 MB) with a bounded retained count (e.g. 5) — fixed defaults, not exposed
  as settings.
- **Format (assumed):** one record per line as **JSONL**
  `{ ts, level, scope, msg, data? }` (machine-greppable + still human-skimmable);
  a text-pretty toggle is out of scope.

## Behavior & states

### Logger
- `log.<level>(scope, msg, data?)` — e.g. `log.info('pty', 'spawn', { sessionId })`.
- A cheap **level gate** runs before any formatting/serialization (so a
  suppressed `trace` costs ~nothing under the firehose).
- **Sinks:** (1) the rotating **file** sink (always, when enabled); (2) the dev
  **console** in development only (`!app.isPackaged`), so packaged builds stay
  quiet on stdout. Sink writes are **best-effort and never throw** into callers.
- **Disabled (`off`):** no file writes, no console; calls become no-ops past the
  gate.

### Lifecycle / config
- Logger is constructed at the earliest point in `main.ts` (before window
  creation) so startup is captured; the level is read from settings and updated
  live when the user changes it (no restart).
- The renderer logs through the existing `log` IPC channel (extended to carry
  `level` + `scope`); the **host** is the single writer to disk (one rotation
  owner, no cross-process file contention).

### States
`disabled` · `enabled@level` · `rotating` (size cap hit → roll file) ·
`write-degraded` (a write failed → drop to console-only / in-memory, never crash).

## Data / interface contract

- `LogLevel = 'off' | 'error' | 'warn' | 'info' | 'debug' | 'trace'` (ordered).
- `LogRecord = { ts: number; level: Exclude<LogLevel,'off'>; scope: string; msg: string; data?: Record<string, unknown> }`.
- **Pure core (testable, no I/O):**
  - `levelEnabled(current: LogLevel, msg: LogLevel): boolean` — ordering gate.
  - `formatRecord(r: LogRecord): string` — JSONL line.
  - `redact(data): data` — strip/mask known-sensitive keys (token/secret/password/
    authorization/cookie/env) before serialization.
  - rotation decision: `shouldRotate(bytesWritten, cap)` and
    `pruneOldLogs(files, keep)`.
- **Settings additions:** `logging: boolean` (enable) + `logLevel: LogLevel`
  (validated/coerced like the rest of `src/settings.ts`).
- **Protocol:** extend `{ type: 'log'; message }` →
  `{ type: 'log'; level?; scope?; message; data? }` (back-compatible: a bare
  `message` defaults to `info`/`scope:'renderer'`).
- **New host IPC:** `revealLogs` (open the logs folder via `shell.openPath`) and
  `copyDiagnostics` (zip/concat the current logs + app/OS version into a bundle
  path, then reveal it).

### Invariants
- A logging failure NEVER crashes or blocks the app (swallow + degrade).
- The PTY byte stream is never logged (only spawn/exit/resize/dispose events).
- Redaction runs before anything reaches a sink.

## Edge cases & failure modes

- **High volume:** terminal output, file watch noise, git-refresh ticks — gate at
  `debug`/`trace`, never `info`; never log raw `term:data`.
- **Write failure / disk full / userData unwritable:** catch, fall back to
  console-only (dev) or an in-memory ring, set `write-degraded`; retry lazily.
- **Rotation race:** the host is the sole writer; rotation is synchronous within
  the write path so two writes can't both roll.
- **Secrets/PII:** paths and repo data are user content (kept), but env/token-like
  keys are redacted; never serialize `process.env` wholesale.
- **Level change mid-run:** applied live; in-flight records honor the gate at emit
  time.
- **Packaged vs dev:** console sink only in dev; file sink in both.
- **Renderer crash/early error:** `logToHost` errors (already wired) must funnel
  through the leveled channel; pre-bridge renderer errors fall back to console.
- **E2E:** under `CONDUIT_E2E`, default to a temp logs dir (or a quieter level) so
  the smoke suite doesn't write into the real userData or spew.

## Defaults vs. settings

- **Exposed:** `logging` (enable, default ON) and `logLevel` (default `info`).
- **Fixed (not exposed):** file location, rotation size, retained count, format,
  redaction key list, console-in-dev-only. Rationale: operational knobs, not
  durable user preferences — exposing them is clutter.

## Scope slicing

**MVP — Slice A:** pure core (`levelEnabled`/`formatRecord`/`redact`/rotation) +
host logger with file sink + rotation; Settings (enable + level, live);
`revealLogs` action; extend the `log` channel; instrument the **top seams**
(app/window lifecycle, session/PTY spawn-exit-dispose, IPC errors, file mutate,
git actions, updater, scrollback persist/restore, second-instance/OS-open).
*Verifiable:* flip level → records appear/disappear in the file; rotation rolls
at the cap; an induced error is captured; reveal opens the folder.

**v1 — Slice B:** renderer-side `log.<level>` parity (not just error funneling);
`copyDiagnostics` bundle (logs + versions); broaden instrumentation coverage; a
"recent log tail" surfaced in Settings → About for quick copy.

**Vision:** in-app log viewer panel (live tail, filter by level/scope, search);
export/share; structured fields surfaced as a table. **Out of scope:** telemetry,
off-machine shipping, logging the PTY stream.

## Acceptance criteria

- **WHEN** `logLevel` is `info`, a `log.debug(...)` call **SHALL** produce no file
  output, and a `log.info(...)`/`log.warn(...)`/`log.error(...)` **SHALL** each
  append one JSONL record to the active log file.
- **WHEN** the active file exceeds the size cap, the logger **SHALL** roll to a
  new file and prune to the retained count, losing no in-flight record.
- **WHEN** logging is `off`, no file is written and no console output occurs.
- **WHEN** a record's `data` contains a token/secret-like key, the persisted line
  **SHALL** show it masked.
- **WHEN** a sink write throws, the app **SHALL** continue normally (no crash, no
  unhandled rejection).
- The "Reveal logs" action **SHALL** open the logs folder in the OS file manager.

```gherkin
Scenario: Dialing the level changes what gets captured
  Given logging is enabled at "info"
  When the app spawns a session (logged at info) and ticks a git refresh (logged at debug)
  Then the log file contains the session-spawn record
  And the log file does not contain the git-refresh record
  When I set the level to "debug"
  And another git refresh ticks
  Then the log file now contains a git-refresh record
```
